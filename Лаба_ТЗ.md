# Лабораторная работа: Meowle Network Lab

## Что вы получаете

В папке лежит APK мобильного приложения:

- `Meowle-lab.apk`

Приложение подключено к учебному серверу Meowle. Работать можно только с этим стендом и только в рамках задания.

## Адреса стенда

- Frontend: http://144.31.255.0:8080
- Backend API: http://144.31.255.0:3001
- Swagger UI / документация API: http://144.31.255.0:3001/api-docs-ui/
- Version check: http://144.31.255.0:3001/version

Swagger UI открывается в браузере и показывает доступные методы API. Если отдельные адреса вроде `/api-docs` или `/swagger.json` не открываются, используйте именно ссылку Swagger UI выше.

## Необходимые инструменты

- Android-телефон или Android Emulator.
- Charles Proxy или Proxyman.
- Браузер.
- Любой инструмент для ручных HTTP-запросов: Postman, Insomnia, curl или встроенный replay в прокси.
- По желанию: Android Studio Logcat.

## Установка приложения

1. Установите `Meowle-lab.apk` на Android-устройство.
2. Если Android блокирует установку, разрешите установку из неизвестных источников для файлового менеджера или браузера.
3. Запустите приложение.
4. На экране входа используйте валидные данные:
   - имя: `Алексей`
   - телефон: `+79990000000`

После входа откройте вкладку рейтинга. Если там отображаются имена котиков, приложение успешно подключилось к серверу.

## Настройка прокси

Для Charles или Proxyman:

1. Убедитесь, что телефон и компьютер с прокси находятся в одной сети.
2. Включите прокси на компьютере.
3. Узнайте IP компьютера в локальной сети.
4. На телефоне откройте настройки текущей Wi-Fi сети.
5. Укажите ручной HTTP proxy:
   - host: IP компьютера;
   - port: порт Charles/Proxyman, обычно `8888` или `9090`.
6. Разрешите подключение телефона в Charles/Proxyman, если появится запрос.

Текущий стенд использует HTTP, поэтому сертификат для базового перехвата не обязателен. Но приложение собрано так, чтобы доверять пользовательским CA-сертификатам, поэтому можно установить сертификат Charles/Proxyman и сравнить поведение HTTP/HTTPS в дополнительных экспериментах.

## Часть 3. Изменение запросов

Выполните минимум 6 экспериментов с изменением запросов через прокси, Postman/Insomnia/curl или replay:

1. Измените `catId` в URL карточки, лайка или дизлайка.
2. Повторите один и тот же запрос лайка несколько раз.
3. Измените описание записи, которую вы не создавали.
4. Попробуйте добавить дубликат имени.
5. Отправьте невалидные значения `gender`, `name`, `catDescription`.
6. Попробуйте вызвать web API-методы, которых нет в основном мобильном сценарии.
7. Проверьте, что происходит при обращении к несуществующему `catId`.
8. Проверьте загрузку файла: корректное изображение, файл с неправильным расширением, файл с неправильным MIME type.

Для каждого эксперимента зафиксируйте:

- исходный запрос;
- изменённый запрос;
- ответ сервера;
- изменилось ли состояние данных;
- является ли поведение багом или security-проблемой;
- как бы вы исправили проблему.

## Часть 4. Изменение ответов

Через Charles Rewrite/Map Local или аналог в Proxyman измените ответы сервера:

1. Подмените имя или описание котика в ответе поиска.
2. Подмените количество лайков/дизлайков.
3. Подмените список результатов поиска.
4. Подмените список фото или URL изображения.

Сделайте скриншоты до и после. Объясните, какие изменения остались только на клиенте, а какие реально сохранились на сервере.

## Часть 5. Security-анализ

Найдите и опишите минимум 6 проблем. Возможные направления проверки:

- авторизация и права доступа;
- прямой доступ к объектам по `id`;
- возможность менять чужие данные;
- возможность повторять действия напрямую через API;
- валидация параметров;
- корректность HTTP-статусов;
- обработка ошибок;
- загрузка файлов;
- rate limit;
- различие между ограничениями в UI и проверками на сервере.

Для каждой проблемы используйте формат:

- Title.
- Severity: Low / Medium / High / Critical.
- Preconditions.
- Steps to reproduce.
- Actual result.
- Expected result.
- Evidence: скриншот, curl, request/response из прокси.
- Recommendation.

## Часть 6. Финальный отчёт

Сдаётся один PDF или Markdown-документ. В отчёте должны быть:

1. Схема стенда: телефон, прокси, сервер, Swagger.
2. Таблица API из Swagger.
3. Карта действий приложения и HTTP-запросов.
4. Минимум 6 экспериментов с изменением запросов.
5. Минимум 4 эксперимента с изменением ответов.
6. Минимум 6 security findings.
7. Минимум 5 баг-репортов.
8. Краткий вывод: почему нельзя доверять мобильному клиенту и какие проверки должны быть на сервере.

## Критерии оценки

- 20%: полнота таблицы API и понимание Swagger.
- 20%: корректный перехват трафика и карта действий приложения.
- 25%: эксперименты с изменением запросов/ответов.
- 25%: качество баг-репортов и security findings.
- 10%: аккуратность отчёта, доказательства, воспроизводимость шагов.

## Ограничения

- Не атакуйте чужие серверы и адреса.
- Не запускайте массовое сканирование, brute force или DoS.
- Не удаляйте данные других групп без необходимости для задания.
- Все эксперименты должны быть воспроизводимыми и описанными в отчёте.
